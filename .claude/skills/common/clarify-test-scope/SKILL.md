# Clarify Test Scope
## この skill を使う場面
- テスト対象が不明確な場合
- テスト種別 (unit/integration/E2E) の判断に迷う場合
## 入力前提
- 実装対象の機能が特定済み
## 実行手順
### Step 1: テスト対象の分類
| 対象 | テスト種別 | 理由 |
|------|-----------|------|
| 純粋関数・ロジック | unit | 外部依存なし |
| Service + DB 連携 | integration | DB 操作を含む |
| ユーザー操作フロー | E2E | 複数層を跨ぐ |
### Step 2: 境界条件の特定
- 正常系 / 異常系 / エッジケース
### Step 3: モック戦略の決定
- unit: 全外部依存をモック
- integration: DB は実物、外部 API はモック
- E2E: 全サービスを実起動
## 判断ルール
- unit で書けるものは unit に寄せる（実行速度重視）
- DB 操作を含む場合は integration
- ユーザーフローの確認は E2E
## 出力形式
テストスコープ定義書 (テスト対象 × テスト種別のマトリクス)
## 注意点
- E2E テストの過多を避ける
## 失敗時の扱い
- スコープが広すぎる場合: 優先度で絞る
