# Clarify Product Requirements

## この skill を使う場面
- ユーザーからの新規機能リクエストが曖昧な場合
- 既存要件の範囲を明確にしたい場合
- 要件間の矛盾を解消したい場合

## 入力前提
- ユーザーからの要望（口頭、テキスト、スクリーンショット等）
- 既存の要件ドキュメント（あれば）

## 実行手順

### Step 1: 要望の初期整理
ユーザーの要望を以下の項目に分類する:
- **誰が**: 対象ユーザー (ペルソナ)
- **何を**: 具体的な機能
- **なぜ**: ビジネス価値・ユーザー価値
- **いつ**: 時間的制約

### Step 2: 質問リストの作成
不明確な点を洗い出し、以下のカテゴリで質問を作成する:
1. スコープ: 含めるもの / 含めないもの
2. エッジケース: 異常系の期待動作
3. 非機能要件: パフォーマンス、セキュリティ、可用性
4. 依存関係: 前提となる既存機能

### Step 3: 要件の構造化
```markdown
## 要件: REQ-XXX [要件名]
- **優先度**: Must / Should / Could / Won't
- **概要**: 一文で説明
- **ユーザーストーリー**: 〜として、〜したい。なぜなら〜。
- **受け入れ条件**:
  - Given: 前提条件
  - When: 操作
  - Then: 期待結果
- **除外事項**: この要件に含めないもの
```

### Step 4: 要件の検証
- 各要件が独立してテスト可能か確認
- 受け入れ条件が tdd-coach にテストケースとして渡せる粒度か確認
- 矛盾する要件がないか確認

## 判断ルール
- 曖昧さが1つでも残る場合は Step 2 に戻る
- 要件が10個以上になる場合は MVP 判断のため define-mvp skill に移行
- AI 機能に関する要件は clarify-ai-requirements skill に委譲

## 出力形式
Markdown 形式の要件ドキュメント。各要件に一意の ID を付与。

## 注意点
- ユーザーの言葉をそのまま使い、解釈の齟齬を避ける
- 技術的な解決方法は要件に含めない（How ではなく What を定義）
- 受け入れ条件は検証可能な形で記述する

## 失敗時の扱い
- ユーザーが質問に答えられない場合: 類似事例を提示して選択させる
- 要件が矛盾する場合: トレードオフを明示して優先度を判断してもらう
