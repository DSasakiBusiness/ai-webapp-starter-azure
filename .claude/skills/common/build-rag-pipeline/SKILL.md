# Build RAG Pipeline

## この skill を使う場面
- ドキュメントに基づいた質問応答機能を追加する場合
- 外部知識を LLM の回答に統合する場合

## 入力前提
- 対象ドキュメントの種類と量が把握済み
- Document / DocumentChunk のスキーマが Prisma に定義済み

## 実行手順

### Step 1: ドキュメント取り込み
1. ドキュメントを受け取る (テキスト、PDF、Markdown)
2. テキスト抽出
3. チャンキング（分割）
   - デフォルト: 500 トークン / チャンク
   - オーバーラップ: 50 トークン
4. DB に保存 (Document + DocumentChunk)

### Step 2: 埋め込み生成
1. 各チャンクに対して AiService.generateEmbedding() を呼び出す
2. 埋め込みベクトルを DocumentChunk.embedding に保存
3. バッチ処理でレート制限を回避する

### Step 3: 類似検索
1. ユーザーの質問を埋め込みに変換
2. コサイン類似度で上位 K 件のチャンクを取得（デフォルト K=5）
3. 類似度スコアの閾値を設定（デフォルト 0.7）

### Step 4: コンテキスト注入
1. 取得したチャンクをシステムプロンプトに注入
2. 引用元情報を含める
3. LLM にコンテキスト付きで質問を送信

### Step 5: レスポンス生成
1. LLM のレスポンスを構造化
2. 引用元（チャンクID、ドキュメント名、ページ番号）を付与
3. ユーザーに返却

## 判断ルール
- チャンクサイズはドキュメントの性質に応じて調整（表形式データは行単位）
- 検索結果が閾値以下の場合は「関連情報が見つかりませんでした」を返す
- 将来的には pgvector 導入を検討する（現在は JSON 保存の暫定実装）

## 出力形式
- `apps/api/src/ai/rag/` 配下のサービス
- 取り込み・検索・生成の各モジュール

## 注意点
- 大量ドキュメントの一括処理はバッチで行う
- 埋め込み API のレート制限に注意
- チャンクの品質がRAGの精度を大きく左右する

## 失敗時の扱い
- 検索結果が空の場合: LLM の一般知識で回答し、「ドキュメントに該当情報なし」を明記
- 埋め込み生成エラー: リトライ後、失敗チャンクをスキップしてログに記録
- 引用が不正確な場合: チャンクサイズとオーバーラップを調整
