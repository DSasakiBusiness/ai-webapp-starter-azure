# Define MVP

## この skill を使う場面
- 要件が多すぎてスコープを絞りたい場合
- 初期リリースに含める機能を決定したい場合

## 入力前提
- clarify-product-requirements で構造化された要件リスト

## 実行手順

### Step 1: MoSCoW 分類
各要件を以下に分類する:
- **Must**: なければ価値がない（必須）
- **Should**: あるべきだが初期リリースでなくてもよい
- **Could**: あれば嬉しい
- **Won't**: 今回は対象外

### Step 2: 技術リスク評価
Must 要件について技術リスクを L/M/H で評価する。H の場合は solution-architect に相談。

### Step 3: MVP スコープ確定
Must 要件 + 技術リスクが低い Should 要件 = MVP として定義。

### Step 4: 実装順序の決定
MVP 内の機能を依存関係に基づいて順序付けする。

## 判断ルール
- Must 要件が 5 個以上の場合は更に優先度を付ける
- 技術リスクが H の Must 要件は PoC を先行する
- AI 機能は段階的にリリースする（初期はシンプルなチャット、次にRAG）

## 出力形式
MVP 定義書: 含まれる要件ID、除外要件とその理由、実装順序。

## 注意点
- 「全部 Must」は MVP ではない
- ユーザーフィードバックを早く得られる構成を優先する

## 失敗時の扱い
- 優先度が決まらない場合: ユーザー価値と技術リスクの 2x2 マトリクスで判断
