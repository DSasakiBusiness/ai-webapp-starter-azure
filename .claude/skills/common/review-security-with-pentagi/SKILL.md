# Review Security with PentAGI

## この skill を使う場面
- リリース前のセキュリティチェック
- 新しい API エンドポイント追加後の検証
- 定期的なセキュリティ監査

## 入力前提
- PentAGI が利用可能（PENTAGI_API_URL, PENTAGI_API_KEY が設定済み）
- ステージングまたは隔離環境がデプロイ済み
- **対象が本番環境でないことが確認済み**

## ⚠️ 安全上の注意（厳守）
1. **本番環境に対して絶対にスキャンを実行しない**
2. スキャン開始前に対象 URL が本番でないことを二重確認する
3. 結果は必ず人間が再確認してから修正に着手する
4. 自動修正を行わない

## 実行手順

### Step 1: 環境確認
```bash
# 対象 URL が本番でないことを確認
echo "Target: $PENTAGI_TARGET_URL"
# 本番 URL と一致しないことを確認
```

### Step 2: スキャン設定
対象とする検査項目:
- API 認証バイパス
- 入力バリデーション (XSS, SQLi)
- ファイルアップロード脆弱性
- セッション管理
- CORS 設定
- HTTP セキュリティヘッダー
- LLM プロンプトインジェクション

### Step 3: スキャン実行
PentAGI のスキャン API を呼び出す。スキャン完了を待機。

### Step 4: 結果分析
各脆弱性を以下で分類:
- **Critical**: 即座に修正必須（認証バイパス、データ漏洩）
- **High**: リリース前に修正必須（XSS、SQLi）
- **Medium**: 次スプリントで修正（セキュリティヘッダー不足）
- **Low**: issue に起票して追跡（情報漏洩の軽微なリスク）

### Step 5: 再現確認
各脆弱性を手動で再現し、誤検知を除外する。

### Step 6: レポート作成
```markdown
## セキュリティレビュー結果
- 日付: YYYY-MM-DD
- 対象: [ステージング URL]
- ツール: PentAGI

### 検出事項
#### [Critical] 脆弱性名
- 影響: ...
- 再現手順: ...
- 修正方針: ...
- 再検証手順: ...
```

## 判断ルール
- Critical / High がある場合はリリースをブロックする
- 誤検知は「誤検知」としてレポートに残す
- 修正は該当する developer agent に委譲する

## 出力形式
セキュリティレビューレポート (Markdown)

## 失敗時の扱い
- PentAGI 接続エラー: setup-pentagi-scan skill で再設定
- 本番環境を対象にしようとした場合: **即座に中止**
- 結果が膨大な場合: Critical/High のみ先に対応し、残りはissueに分割
