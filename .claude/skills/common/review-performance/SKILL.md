# Review Performance
## この skill を使う場面
- パフォーマンス問題の調査時
- リリース前のパフォーマンス確認
## 入力前提
- パフォーマンス目標値が定義済み (レスポンスタイム等)
## 実行手順
### Step 1: 計測
- API レスポンスタイム (目標: p95 < 500ms、AI 機能は < 10s)
- フロントエンド LCP (目標: < 2.5s)
- DB クエリ時間
### Step 2: ボトルネック特定
- 遅いクエリの特定 (Prisma logging)
- N+1 クエリの検出
- メモリリークの確認
### Step 3: 改善策の提案と実装
- クエリ最適化 (include/select の見直し)
- インデックス追加
- キャッシュ層の導入
### Step 4: 改善後の再計測
## 判断ルール
- p95 レスポンスタイムが目標を超える場合は改善必須
- AI 機能は LLM の応答時間を含むため長めの目標を設定
## 出力形式
パフォーマンスレポート (計測値 + 改善策)
## 注意点
- 計測は本番に近い環境で行う
- 開発環境の数値を鵜呑みにしない
## 失敗時の扱い
- 改善が困難な場合: solution-architect にアーキテクチャ変更を相談
