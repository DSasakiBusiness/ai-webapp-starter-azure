# Integrate LLM Feature

## この skill を使う場面
- 新しい AI 機能を追加する場合
- 既存の LLM 統合を拡張する場合

## 入力前提
- AI 機能の要件が clarify-ai-requirements で明確化済み
- AiService が `apps/api/src/ai/ai.service.ts` に存在

## 実行手順

### Step 1: プロンプト設計
- システムプロンプトを定義（役割、制約、出力形式）
- 入力変数を特定
- 出力形式を定義（テキスト / JSON / 構造化）

### Step 2: TDD で実装
1. AiService のモックを使ったテストを書く
2. 期待する入出力構造を定義
3. エラーケース（API エラー、タイムアウト、レート制限）のテストを書く
4. 実装して Green にする

### Step 3: エラーハンドリング
- API エラー: リトライ（最大 3 回、指数バックオフ）
- レート制限 (429): キューイングまたはフォールバック
- タイムアウト: 30 秒で打ち切り、ユーザーに通知
- 不正な出力: ガードレール（バリデーション）を追加

### Step 4: コスト管理
- トークン使用量をレスポンスに含める
- 入力トークン数の上限を設定
- モデルの選択基準を文書化

## 判断ルール
- シンプルなタスクには安価なモデル (gpt-4o-mini) を使用
- 複雑な推論には gpt-4o を使用
- Azure OpenAI が設定されている場合はそちらを優先
- プロンプトインジェクション対策を必ず含める

## 出力形式
- `apps/api/src/ai/` 配下のサービス・コントローラー
- プロンプトテンプレート

## 注意点
- API キーをハードコードしない
- プロンプトにユーザー入力を直接埋め込まない（テンプレート経由）
- レスポンスの構造を検証してからクライアントに返す

## 失敗時の扱い
- LLM が期待外の出力を返す場合: ガードレールで検出し、リトライまたはフォールバック
- コストが予想を超える場合: トークン上限を下げ、モデルを変更
